Deployment & CI/CD

Every repo deploys to Cloudflare Pages via a centralised reusable workflow in ProDocStore. Each repo's docs-deploy.yml is just a thin caller - all deployment logic, Access protection, and rollback lives in one place.

Publishing a New Site - Step by Step

1. Create docs/ and add content

mkdir docs
cp prodocstore/docs/index.html docs/index.html
# Edit title, badge text, sidebar nav, and content

2. Create docs-deploy.yml - just set your project name

mkdir -p .github/workflows
cp prodocstore/templates/workflows/docs-deploy.yml .github/workflows/

Then edit docs-deploy.yml - only one value to change:

name: Docs Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    uses: ProDocStore-online/platform/.github/workflows/deploy-pages.yml@main
    with:
      project-name: your-project-name
    secrets: inherit

3. Push to main

git add docs/ .github/workflows/
git commit -m "Add docs site and docs-deploy workflow"
git push origin main
# GitHub Actions deploys automatically
# Access protection is created automatically on first deploy

Feature Flags (docs/features.json)

Deploy-time add-ons (changelog, sitemap, nav, search, toc, codeBlocks, etc.) are toggled via a single docs/features.json per KB. Drop the file in, list the add-ons you want on, and the reusable workflow reads it at deploy time. Missing keys fall through to the corresponding workflow input (legacy KBs that haven't migrated yet keep working). Schema: templates/features.schema.json in ProDocStore; canonical catalog with descriptions: templates/add-ons.json (read by the chat extension's system prompt and Options UI).

{
  "$schema": "https://raw.githubusercontent.com/ProDocStore-online/platform/main/templates/features.schema.json",
  "changelog": true,
  "sitemap": true,
  "pageMeta": true,
  "references": false,
  "nav": true,
  "search": true,
  "toc": true,
  "codeBlocks": true,
  "callouts": true,
  "prevNext": true,
  "branding": {
    "logo": "logo.svg",
    "accent": "#d85c42",
    "lineColor": "#d8cdbb"
  }
}

Why one JSON per KB instead of flags in each repo's docs-deploy.yml? Discoverability (one file tells you what this site does), git history that's not buried in CI syntax, and the same file can later drive runtime behaviour if we want (e.g., the chat extension reading it to adapt). Secrets and CI-only concerns (Access email lists, office CIDRs) stay in YAML where they belong.

How to enable a feature

Three routes, pick whichever fits your workflow:

Via chat (fastest)

Open the ProDocStore side panel on your KB, switch to Edit mode, and prompt:

turn on sitemap and pageMeta

The agent reads docs/features.json, toggles the keys, and shows a preview. Click Apply to open a PR (or push directly, per your commit-mode setting).

Edit docs/features.json by hand

{
  "sitemap": true,
  "pageMeta": true
}

Commit + push. Cloudflare Pages redeploys within ~30 s and the feature kicks in. Keys you don't set fall through to the defaults in your docs-deploy.yml.

How to request a NEW add-on

An add-on not in the table above is a ProDocStore change, not a per-KB change. New add-ons need coordinated edits across multiple files: an entry in templates/add-ons.json (catalog), a property in templates/features.schema.json, a generator script in templates/search/scripts/, a gated step in .github/workflows/deploy-pages.yml, and tests in extension/tests/. File an issue on ProDocStore-online/platform describing the add-on + the shape of its output, and either open a PR or ask the ProDocStore agent (Edit mode) to scaffold it. inject-toc.mjs and inject-codeblocks.mjs are the cleanest reference implementations to copy from.

Secure-by-Default Deployment

All ProDocStore sites use a secure-by-default deployment workflow. No site is ever publicly accessible - the workflow enforces this with a 4-step guarantee:

StepWhat It DoesIf It Fails
1. Ensure AccessCreates/verifies Cloudflare Access app + @your-company.com policyDeployment blocked
2. Pre-flightCurls the live URL - must NOT return HTTP 200 (public)Deployment blocked
3. DeployPushes to Cloudflare Pages via WranglerWorkflow fails
4. Post-deployCurls the live URL again - must NOT return HTTP 200Deployment rolled back (deleted)
Hard rule

If anything fails, the site stays down. A broken deploy is better than a public deploy. All four steps live in deploy-pages.yml in this repo - no repo can skip or override them. To change deployment behaviour, update the reusable workflow once and every repo picks it up automatically.

How it works

Pre-flight check

Before deploying, the workflow curls the site. If it returns 200 (publicly accessible), the deploy is blocked. Expected responses:

  • 302/303 - Access login redirect (protected, good)
  • 000 - Site doesn't exist yet (first deploy, safe)
  • 4xx/5xx - Site is down (safe to proceed)

Post-deploy rollback

After deploying, the workflow curls the site again. If it returns 200, the latest deployment is automatically deleted via the Cloudflare API, taking the site down immediately.

How Reusable Workflows Work

GitHub Actions supports reusable workflows - a workflow in one repo that other repos can call like a function. This is how every repo deploys with a single line of config while all logic lives in ProDocStore.

The Concept

Reusable Workflow (callee)

Lives in ProDocStore at .github/workflows/deploy-pages.yml. Declares on: workflow_call instead of on: push. Defines inputs (like function parameters) and runs all the actual deployment logic.

# deploy-pages.yml (in prodocstore)
on:
  workflow_call:
    inputs:
      project-name:
        required: true
        type: string
      deploy-dir:
        required: false
        type: string
        default: "docs"

Caller Workflow

Lives in each project repo at .github/workflows/docs-deploy.yml. A thin wrapper that triggers on push and calls the reusable workflow with a project name.

# docs-deploy.yml (in each repo)
on:
  push:
    branches: [main]

jobs:
  deploy:
    uses: ProDocStore-online/platform/.github/workflows/deploy-pages.yml@main
    with:
      project-name: your-pages-project
    secrets: inherit

How workflow_call Works

ConceptWhat It MeansOur Usage
on: workflow_callMakes a workflow callable by other workflows (like exporting a function)deploy-pages.yml declares this trigger
uses: owner/repo/.github/workflows/file.yml@refCalls a reusable workflow from another repo at a specific git refEach repo calls ProDocStore-online/platform/...@main
inputs:Typed parameters the caller must/can provide (string, boolean, number)project-name (required), deploy-dir, fetch-brand-assets, etc.
secrets: inheritPasses all of the caller's secrets to the reusable workflow automaticallyPasses CLOUDFLARE_API_TOKEN and CLOUDFLARE_ACCOUNT_ID from org secrets
@main refAlways uses the latest version on the main branchEvery repo gets updates automatically when changes land in ProDocStore
Why @main instead of tags

Repos referencing @main - this means any push to deploy-pages.yml in ProDocStore immediately affects every repo's next deploy. This is convenient but risky: a broken commit breaks all deploys. Issue #6 tracks moving to tagged releases (@v1, @v2) so repos can pin to a known-good version and upgrade deliberately.

How secrets: inherit Works

When a caller workflow specifies secrets: inherit, GitHub automatically passes all secrets available to the caller into the reusable workflow. For ProDocStore repos:

  1. Org-level secrets (CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID, SLACK_TOKEN, SLACK_CHANNEL) are set on the ProDocStore GitHub organisation and available to all repos
  2. Repo-level secrets (ANTHROPIC_API_KEY) are set per-repo and only available in that repo
  3. The reusable workflow accesses them via ${{ secrets.CLOUDFLARE_API_TOKEN }} - same syntax as normal workflows

The alternative is declaring each secret explicitly in the workflow_call definition, but inherit is simpler when you trust the caller.

Cross-Repo Permissions

For uses: ProDocStore-online/platform/...@main to work:

  • The ProDocStore repo must be internal or public within the org (private repos cannot share reusable workflows unless you enable "Accessible from repositories in the organisation" under Actions settings)
  • The caller repo's Actions settings must allow workflows from ProDocStore
  • The permissions: block in the reusable workflow defines what GitHub token scopes it needs (contents: read, deployments: write)

What the Reusable Workflow Actually Does

The deploy-pages.yml workflow runs these steps in order, all in a single job:

#StepWhat HappensCondition
1CheckoutChecks out the caller repo's code (not ProDocStore)Always
2Fetch brand assetsChecks out ProDocStore via sparse checkout and copies styles.css, logo.svg, favicon.svg, nav.css, robots.txt, _headers into deploy dirAlways (default)
3Pre-deploy buildRuns each command from pre-deploy-steps JSON array via jq -r '.[]' piped to evalpre-deploy-steps != '[]'
4Ensure AccessCalls Cloudflare API to find or create an Access self-hosted app + email domain policyAlways
5Pre-flight checkCurls https://{project}.pages.dev - blocks if HTTP 200 (publicly accessible)Access is ready
6Create Pages projectRuns wrangler pages project create if this is first deployFirst deploy only
7DeployUses cloudflare/wrangler-action@v3 to run wrangler pages deployAlways
8Create Access app (first deploy)Now that Pages project exists, creates the Access app + policy. Rolls back deploy if this fails.First deploy only
9Post-deploy verifyCurls the live URL again - if HTTP 200, deletes the deployment via APIAlways

The pre-deploy-steps Input

This input accepts a JSON array of shell commands that run before deployment. The workflow parses it with jq and executes each command via eval:

# In the workflow:
echo '${{ inputs.pre-deploy-steps }}' | jq -r '.[]' | while IFS= read -r cmd; do
  echo "Running: $cmd"
  eval "$cmd"
done
Security note: eval

The use of eval here means the caller can execute arbitrary shell commands in the runner. This is acceptable because callers are trusted (they're our own repos), but it's flagged in Issue #1 for replacement with a safer mechanism like composite actions or a matrix strategy.

Updating the Workflow

Because all repos use @main, updating the deployment process is a single commit:

  1. Edit .github/workflows/deploy-pages.yml in ProDocStore
  2. Push to main
  3. Every repo's next deploy picks up the change automatically

No repo needs to be touched. This is the main advantage of the reusable workflow pattern - one change, all repos updated.

Quality & Safety

Two quality gates run on every change to ProDocStore and every deploy from a project repo: the compliance lint and the unit test suite. Both are enforced automatically - there is no way to accidentally ship a broken generator or a non-compliant site.

Compliance Lint

The lint runs as the first step of every deploy (in deploy-pages.yml). If any check fails, the deploy is blocked before any Cloudflare side-effects happen. Logic lives in templates/search/scripts/docs-lint.sh so it is testable and also runnable locally.

RuleSeverityRationale
Workflow must be named docs-deploy.yml, not deploy.ymlFAILAvoids clash with project app deploy workflows
No brand assets tracked in docs/ (styles.css, logo.svg, favicon.svg, nav.css, robots.txt, _headers)FAILThese are injected by the workflow at deploy time
No publishing scripts in top-level scripts/ (generate-*, inject-*, add-heading-ids)FAILGenerators live centrally in ProDocStore
Every HTML page has <meta name="robots" content="noindex,…">FAILSites are internal - never index
Missing <meta name="googlebot">WARNOptional - robots already covers Google
Pages with standard topbar link to styles.cssFAILShared stylesheet is the source of truth; print-style pages with @page or *-pdf.html are exempt
Pages with standard topbar link to favicon.svgFAILTabs should show the site icon
No financial/contractual files (sow, proposal, estimate, rate-card, pricing, quote, invoice, contract)FAILOnly permitted in dedicated commercial sites via allow-financial: true

Opt-out flags (use sparingly)

allow-inline-styles: trueLegacy migration escape hatch; not needed today

Cloudflare Pages Deep Dive

Every ProDocStore documentation site is hosted on Cloudflare Pages - Cloudflare's static site hosting platform. No servers, no containers - just static files served from Cloudflare's global edge network.

How a Pages Project Gets Created

On first deploy, the reusable workflow creates the Pages project automatically:

npx wrangler pages project create "$PROJECT_NAME" --production-branch main

This creates an empty project in the Cloudflare account. Subsequent deploys push files into it:

wrangler pages deploy docs/ --project-name=your-pages-project

The cloudflare/wrangler-action@v3 GitHub Action handles authentication using CLOUDFLARE_API_TOKEN and CLOUDFLARE_ACCOUNT_ID.

How Wrangler Deploys Work

  1. Wrangler hashes every file in the deploy directory
  2. Only uploads files that have changed (content-addressable storage)
  3. Creates a new deployment pointing to the uploaded files
  4. The new deployment is instantly live on the .pages.dev URL
  5. Previous deployments are retained (you can roll back in the Cloudflare dashboard)

Pages Project Configuration

ProDocStore Pages projects have no build configuration in Cloudflare - all builds happen in GitHub Actions. The Cloudflare dashboard shows:

SettingValueWhy

Rollback Mechanism

When the post-deploy check fails (site returns HTTP 200 = publicly accessible), the workflow deletes the latest deployment via the Cloudflare API:

# Get the latest deployment ID
DEPLOYMENTS=$(curl -s \
  "https://api.cloudflare.com/client/v4/accounts/${ACCOUNT_ID}/pages/projects/${PROJECT}/deployments" \
  -H "Authorization: Bearer ${TOKEN}")
LATEST_ID=$(echo "$DEPLOYMENTS" | jq -r '.result[0].id // empty')

# Delete it - takes the site down immediately
curl -s -X DELETE \
  "https://api.cloudflare.com/client/v4/accounts/${ACCOUNT_ID}/pages/projects/${PROJECT}/deployments/${LATEST_ID}?force=true" \
  -H "Authorization: Bearer ${TOKEN}"

This is a hard delete - the deployment is removed, not just deactivated. The previous deployment (if any) becomes the live version.

Cloudflare Zero Trust & Access

A ProDocStore-deployed .pages.dev site can be protected by Cloudflare Access, part of Cloudflare's Zero Trust platform. When configured this way, the site is not publicly accessible - users must authenticate with a @your-company.com email before seeing any content.

What Is Cloudflare Access

Cloudflare Access is an identity-aware proxy that sits in front of web applications. It checks who you are before any request reaches the origin server (or in our case, the Pages static files).

ConceptWhat It MeansOur Setup
Self-hosted applicationAn Access app that protects a domain you controlOne app per .pages.dev site
PolicyRules that determine who can access the application"Allow if email domain = openfrontier.one"
Session durationHow long a user stays authenticated before re-prompting24 hours
Identity providerHow users prove their identityOne-time PIN (OTP) sent to email

The Authentication Flow

When someone visits a protected site, this is exactly what happens:

  1. User navigates to https://your-project.pages.dev
  2. Cloudflare's edge intercepts the request - checks for a valid CF_Authorization cookie
  3. No valid cookie found → 302 redirect to https://your-team.cloudflareaccess.com/cdn-cgi/access/login/your-project.pages.dev
  4. User sees the Cloudflare Access login page - enters their @your-company.com email
  5. Cloudflare sends a one-time PIN to that email address
  6. User enters the PIN on the login page
  7. Cloudflare verifies the PIN, checks the email domain against the policy (openfrontier.one)
  8. If allowed: sets a signed JWT cookie (CF_Authorization) with a 24-hour TTL
  9. User is redirected back to the original URL - this time the cookie is present, so the request passes through to the Pages content
  10. For the next 24 hours, all requests include the cookie and pass through without re-authentication

How Access Apps Are Created

The reusable workflow creates Access apps automatically via the Cloudflare API. Here's exactly what it does:

Step 1: Check if an Access app already exists

# List all Access apps in the account
curl -s "https://api.cloudflare.com/client/v4/accounts/${ACCOUNT_ID}/access/apps" \
  -H "Authorization: Bearer ${TOKEN}"

# Find one whose self_hosted_domains includes our domain
# e.g., "your-project.pages.dev"
APP_ID=$(... | jq -r '.result[] | select(.self_hosted_domains[]? == $DOMAIN) | .id')

Step 2: Create the app if it doesn't exist

curl -s -X POST "https://api.cloudflare.com/client/v4/accounts/${ACCOUNT_ID}/access/apps" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "your-pages-project",
    "domain": "your-project.pages.dev",
    "type": "self_hosted",
    "session_duration": "24h"
  }'

This registers the domain with Cloudflare Access. From this moment, any request to that domain is intercepted by Access.

Step 3: Add the email domain policy

curl -s -X POST "https://api.cloudflare.com/client/v4/accounts/${ACCOUNT_ID}/access/apps/${APP_ID}/policies" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Allow ProDocStore emails",
    "decision": "allow",
    "include": [{
      "email_domain": {
        "domain": "openfrontier.one"
      }
    }]
  }'

This creates an "Allow" policy that grants access to anyone with an @your-company.com email. All other emails are denied.

Policy Structure

Each Access app has one or more policies. Ours uses a single "Allow" policy with an email domain include rule:

FieldValueMeaning
decisionallowGrant access if the include rules match
includeemail_domain: openfrontier.oneMatch any email ending in @your-company.com

Other policy types exist (Deny, Bypass, Service Auth) but we only use Allow. You can add additional include rules (specific emails, IP ranges, identity provider groups) in the Zero Trust dashboard → Access → Applications → select app → Policies.

The First-Deploy Race Condition

There's a known ~15-second window where a new site could be publicly accessible:

  1. Pages project doesn't exist yet → Access app can't be created (domain must exist first)
  2. Workflow creates the Pages project and deploys content
  3. Site is live and unprotected for ~15 seconds
  4. Workflow creates the Access app immediately after deploy
  5. Site is now protected

This is tracked as Issue #11 and accepted as a known risk because:

  • It only happens on the very first deploy of a brand-new site
  • The URL isn't published or linked anywhere during this window
  • The content is internal documentation, not secrets or credentials
  • If the Access app creation fails, the workflow rolls back the deployment entirely
