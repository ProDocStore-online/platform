The Dash Problem

AI tools are everywhere in our workflow. That's fine. What's not fine is publishing text that reads like it came straight out of a model. This guide covers the tells and how to avoid them.

What Dashes Actually Are

The names come from old metal typesetting, where character widths were measured in ems and ens:

CharacterNameWidth originKeyboardHTML entity
-HyphenNarrowDirect key-
–En dashWidth of the letter NOption+Hyphen (Mac)&ndash;
—Em dashWidth of the letter MOption+Shift+Hyphen (Mac)&mdash;

What Each One Is For

Hyphen (-)

Joins compound words and prefixes:

well-being, client-side, re-evaluate

En dash (–)

Connects ranges and pairs:

2024–2026, 10–15 hours, Sydney–Melbourne

Em dash (—)

Creates a break in thought or aside:

The result was clear—we needed a rewrite.

Why Em Dashes Are a Problem

Em dashes are legitimate punctuation. Plenty of good human writers use them. The issue is that language models use them constantly as default glue for clarifications, contrasts, and asides. When readers see em dashes on every page, in every paragraph, the text starts to feel synthetic.

One em dash in a page is fine. Five is a pattern. Ten is a red flag.

Our Rules

  1. Don't use em dashes in running text. Almost every em dash can be replaced by a full stop, colon, semicolon, comma, or parentheses. Pick whichever fits the sentence.
  2. En dashes are fine in ranges. 2024–2026, 40–60h, pp. 12–15. That's what they're for.
  3. Don't use en dashes as punctuation. If you're tempted to write "the app – which launched last month – is growing", use commas or parentheses.
  4. When you need a light separator and nothing else works, use a hyphen with spaces: the app - now in beta - is growing. This is what people actually type.
  5. In HTML table cells, a plain hyphen - works as an empty-cell placeholder.

Beyond Dashes: Other AI Tells

Em dashes are the most obvious giveaway, but they're a symptom. The real issue is that AI-generated text has a recognisable texture.

Over-consistent rhythm

AI produces sentences with uniform length and cadence. Too many follow the same shape: "X is not Y. It is Z."

Fix: Vary sentence length. Mix short punchy statements with longer ones. Let some paragraphs be one sentence.

Relentless clarity

Everything explained cleanly, no rough edges, no assumptions about the reader. Real writing trusts the reader more.

Fix: Not every term needs defining. If you're writing for developers, write like a developer talking to another developer.

Repetitive rhetorical structures

Watch for these appearing over and over:

  • "not just X, but Y"
  • "the difference between X and Y"
  • "X. Here's why."
  • "Let's break this down."

Fix: Read your text aloud. If you hear the same pattern three times, rewrite the third one differently.

Uniform tone across pages

Human multi-page docs drift a bit between sections or authors. A technical architecture page reads differently from a user-facing overview. AI stays unnaturally even.

Fix: Let technical pages be drier. Let overviews be warmer. Don't sand everything to the same finish.

Overuse of tidy labels

"Design Principles", "Implications", "Why This Matters", "TL;DR", "Key Takeaways" are useful individually. When every page is organised this neatly, it feels templated.

Fix: Use descriptive headings specific to the content. "How BLE discovery works" beats "Technical Overview".

Clean but generic examples

"Short sleep", "no calls today", "30-minute walk" read as plausible but synthetic. They're the kind of examples a model generates without real data.

Fix: Use real examples from the product wherever possible. Specific beats generic.

Minimal messiness

Human drafts have uneven emphasis, slightly inconsistent terminology, occasional shortcuts, and the odd awkward transition. AI removes too much of that.

Fix: Don't over-polish. A doc that reads like it was written by a real person in a real afternoon is more trustworthy than one that reads like it was generated and reviewed by committee.

The Practical Test

Read your page and ask:

  1. Could a reader tell this was AI-assisted just from the writing style?
  2. Do more than two em dashes appear on this page?
  3. Does every paragraph follow the same structure?
  4. Would this sound natural if you read it aloud to a colleague?

If the answer to 1, 2, or 3 is yes, or 4 is no, edit it down. The goal isn't to hide that you used AI. The goal is to publish text that sounds like it was written by someone who cares about the subject, not by a tool that processes tokens.

Publishing Rules

Hard rules for everything we publish across all ProDocStore knowledge bases.

Content format

  • All browsable content must be HTML pages in docs/. Never publish standalone markdown files as site content. Markdown is only for repo-level files that GitHub renders (README.md, CONTRIBUTING.md).
  • Every page uses the shared styles.css, topbar with logo and badge, sidebar navigation, and footer.

No financial or contractual content in project repos

  • Statements of work, proposals, estimates, rate cards, pricing, and any contractual documents must only live in a dedicated commercial site, not in project KBs. Individual project knowledge bases are for technical documentation, architecture, and project knowledge only.
  • This includes SOW HTML pages, proposal PDFs, internal estimates, signed contracts, and any content containing dollar amounts, payment terms, or engagement models.
  • If a project KB currently has this content, it needs to be migrated out and removed from the project repo.

docs/ is self-contained - nothing else publishing-related in the repo

  • Everything related to docs publishing must live inside docs/. No publishing scripts, build tools, generators, stylesheets, or assets outside of it. The repo root belongs to the project's code, not its documentation site.
  • Do not copy build scripts into individual repos. Generators for changelogs, sitemaps, and page metadata are maintained centrally in ProDocStore and run by the reusable workflow at deploy time. Enable them with flags like generate-changelog: true in docs-deploy.yml.
  • The only exception is .github/workflows/docs-deploy.yml, which must live in its standard location to trigger CI.

Writing quality

  • Avoid em dashes in running text. Use full stops, colons, commas, or parentheses.
  • En dashes only in numeric ranges.
  • Vary sentence rhythm. Don't let every paragraph follow the same structure.
  • Use real, specific examples over generic placeholders.
  • Let some human messiness through. Over-polished text reads as generated.
A good target

If a reader notices the punctuation pattern at all, it's probably too much.
